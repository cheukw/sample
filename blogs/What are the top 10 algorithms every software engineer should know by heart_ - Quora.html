Carson McNeil, former Software Engineer - Human Sensing at Google (2013-2016)
Written Feb 13 · Upvoted by
Gideon Shavit, Master's Computer Science, University of Washington

I agree with many of the answerers. I was a Google engineer for three years, did a lot of projects, and was quickly promoted, so I think I'm a pretty solid software engineer. As I read this to myself, I wondered… do I even know ten algorithms by heart, in full detail?? I don't think so.

Understanding algorithms is an invaluable skill, but like many domains of knowledge, the skill is in a way of thinking, not in memorization. You want to be able to read an algorithm, and immediately understand why it is the way it is. As far as knowing exactly how a specific algorithm is implemented, that's what Wikipedia is for.

That being said, here are a couple of the (less than 10) algorithms that I DO know by heart, because they're pretty fundamental:

  1. Binary search. Binary search. Binary search. This is an incredibly important concept, and it pops up in everything from convex optimization to databases. If you need to find a particular value in a big set of things, and you can figure out which direction you need to go, use binary search for a MASSIVE speed up.
  2. Matrix operations. Now, this one only pops up for me because I'm more of a data scientist, but in general, it's important to remember that matrix multiplication is faster than you think it is. Brilliant people have come up incredibly unintuitive algorithms to make matrices multiply in O(n^2.something) instead of O(n^3), like you would expect. In practice, these algorithms aren't necessarily used, but the BLAS and MPI libraries do provide incredibly fast matrix multiplication all the same. Anytime you can express a bunch of calculations in terms of matrix operations, you win.
  3. Linear regression. Simple, beautiful, and useful.
  4. Caching. Just…use caching. Does that count as an algorithm? I don't know, but it is the cause of, and solution to, all of your problems.
  5. The idea of a concurrent work queue.
  6. Knowing when to throw hashtables at a problem.

Glenn Irwin, 30+ years coding, analyzing, designing, architecting, and back to coding
Written Sep 8, 2016

Learning algorithms would be like a chemist memorizing chemical compounds, they are infinite. Here are ten things a programmer should understand to create good algorithms.

A programmer creates algorithms using a finite set of instructions. It’s better to learn the set of instructions and constructs that are used to make algorithms.

  • Read and write from memory/data structures or input/output
  • Mathematical operations (% or modulus is a useful one for programmers)
  • Conditional branching (if, then, elseif, else, switch, etc…
  • Subroutines (to help organize your code and operations)
  • Looping (for, while, etc..)

Algorithms are just assemblies of the above instructions that do something useful. But remember that algorithms are like recipes in that they must be adjusted and adapted to fit your context and taste. The fastest sort algorithm is of no use if it sorts numbers and you are looking sort strings (now you have to convert your string to a number - losing most of the fastness you gained). Like recipes, you can’t jam two of them together and expect it to work, but you can make new ones very easily once you understand the components that make them. This comes only from practice - like cooking.

It also helps to learn some common concepts and patterns used in algorithms (simple algorithms in themselves built from the 5 items above):

  • Sorting ( a simple embedded sort of a few items in a list can be much faster than calling any sort library)
  • Data structures (accessing, manipulating, and navigating stacks, lists, etc…)
  • Control-break loops (do they still teach this? - its useful)
  • Callbacks and embedded functions and different code organizations (OO)
  • Recursion (my personal favourite)

With some practice, you will be writing your own custom algorithms that efficiently do what you want. Don’t try to create a beautiful algorithm straight away, they don’t come that way out of your mind (at least not mine).

Start by just writing code to do what you want to do in the simplest way you can imagine it should work. As you write it and get it to work, you will find you need functions to support something your code needs so you write them and organize your code around them. Then you will find that you have variables that aren’t used - so you take them out, then you will find code that does nothing - so you take it out. Then you will find functions that get called only once in the entire system so you put that code back in to where it is called. Then you find code that doesn’t really belong in your code but is there to let you talk to some other code which also converts the data to a better common format that you can use - so you take out the conversions and change your stuff to use the other format. You may also find configurable items that will never need configuration so you take those out and hard code them so the system can’t be broken by changing a configuration item that isn’t needed. What you are left with is an algorithm except that you find that it’s actually all really simple and should have been obvious from the beginning. You will think it is beautiful but only you because you saw how ugly it could have been. Some people just get it to work and leave it ugly and inefficient and full of ways to break. Good algorithms emerge when we take out the unnecessary code and reduce the solution of a problem into a simple working reliable system that provides value to someone.

Algorithms are not what programmers study, it’s the result of their work. Other than the items above, I don’t have any specific algorithms memorized that I can think of after 25+ years developing software. Maybe ‘software engineers’ are different.

Greg Skinner, 25+ years of software engineering, mostly in C and Perl, mostly in the BSD networking stack and Uni…

None.

Rather than memorizing algorithms, you should understand algorithmic principles, and know those algorithmic principles by heart. The goal in understanding algorithms is to be able to analyze algorithms you haven’t seen before, to be able to apply existing algorithms to new problems, and to be able to create new algorithms from scratch.

Of course, in order to gain a good understanding of those algorithmic principles, you will need to study a lot of algorithms (way more than 10). It may be useful to keep to these algorithms you learn in your mind as canonical examples of the principles they deploy.

For instance, Mergesort is a good canonical example of the Divide-and-Conquer principle. It is not the only canonical example, and I would expect that someone who understands Divide-and-Conquer well would have a few examples in their mind. But someone who knows the Divide-and-Conquer principle behind Mergesort would probably be able to write pseudocode for it or code up an implementation without references. Although if they haven’t done so in a long time, it might take them a little while to produce the correct implementation, and their first attempt might be buggy.

In that sense, yes they do know Mergesort because they understand the principle behind it. However, it’s not like they “know by heart” the actual code. And they should not be expected to code it up without thinking. Quite the opposite, they should code it up by consciously referring in their mind to the principle that it uses. The end result may or may not look like the code they originally saw when they first learned Mergesort but it will use the same principle. Someone who understands algorithms well should never feel the need to memorize how to code any algorithm!!!

Here are 10 algorithmic principles I think that every software engineer should know, roughly in the order of how important I think it is to know them:

  1. How to analyze the running time of non-recursive code (nested for loops, while loops, etc.)
  2. How to create and traverse dynamic linked data structures using pointers, and understanding when those data structures can be implemented as implicit/succinct data structures without using pointers (a good example is the binary heap, which could be implemented with pointers but is usually implemented implicitly for efficiency reasons)
  3. Graph traversal (BFS and DFS) and the various graph properties you might be interested in (connectivity, shortest path, cycle detection, etc.)
  4. How to analyze the running time of recursive code (formulating recurrences, using Master Theorem, recognizing common recursive structures)
  5. The divide and conquer principle (e.g. merge sort, binary search, fast exponentiation, majority algorithm, select/median-finding algorithm, Karatsuba/Toom-Cook multiplication, fast fourier transform, Strassen matrix multiplication, Preparata-Hong convex hull, closest point pair, etc.)
  6. Dynamic programming and memoization (e.g. knapsack, shortest path, CYK parsing algorithm, longest common subsequence, edit distance, Knuth word wrap, solving games, etc.)
  7. Iterative methods and how to analyze their convergence, error-bounding and numerical stability properties. In particular, you should at least be familiar with gradient descent/ascent and some common optimizations like Newton’s method. Should know common applications like root finding and solving linear systems.
  8. Branch-and-bound search strategies and their derivatives used in AI like A*, IDA, alpha-beta pruning, etc.
  9. The general principle of reductions, i.e. taking one problem and modelling it as another. In particular, it is useful to be able to model problems you want to solve as constraint satisfaction problems or linear programs, and use general purpose solvers to get the solution. Furthermore, sometimes you realize the constraints of your problem allow it to be modeled as a problem that can be solved with a specialized algorithm, e.g. modelling as a flow problem or a shortest path problem. The idea of reductions is also used a lot in computational complexity, for instance to prove that a problem is NP-hard, which is useful to a practitioner who does not know the complexity of a particular problem.
  10. Various math tricks from number theory, linear algebra, trigonometry, computational geometry, etc. which pop up frequently in computer science.

Depending upon the kind of work you do, you might re-order the importance of the second half of this list. However, I think it is most important for a good computer scientist or software engineer to have a mastery of all these principles before they try to memorize a single algorithm. Just my 2 cents.

Basically just understanding why an algorithm is considered good and why another is bad is more important than just knowing. Typically run time ( CS focuses on worst case ) is a pretty important factor in the “is this a good algorithm” decision.

But if you need a list of things to start exploring algorithms I would suggest starting with:

  • Set Theory
  • Data Structures (Stack, Heap, Trees, etc.)
  • Sorting Algorithms (Bubble, Quick, Heap Sort, etc.)
    • With a short detour to Huffman Coding
  • Big O Notation (Asymptotic Notation)
  • Divide and Conquer
  • Graph Theory
  • Dynamic Programming
  • Search Algorithms ( Dijkstra's algorithm, Bellman Ford Algorithm, Depth First Search (DFS) , Breadth First Search (BFS), A* (A-Star), etc.)

Note: This is defintitely not a complete list, just the stuff I remember off the top of my head from undergrad that I feel are important (i.e defintiely an opinion list (but what “Top X” list isn’t an opinion list?))

And then the obligatory plug for Introduction to Algorithms if you’re still interested or want a really good resource for algorithms in general

Here is my top ten :

Regression method for predicting the value of a certain value. For example : How much will a given customer use a certain service? The quantity to be predicted here is service usage, and a model could be generated by looking at other, similar individuals in the population and their historical usage.

Clustering :is the problem of grouping the individuals in a population together by their similarity of attributes.

Classification algorithms. Classification tries to predict, for each individual in a population, which set of classes this individual belongs to. A classification task, given a new individual, determines which class that individual belongs to. It also mya assign a probability to this association. An example is KNN (or k neares neighbour ) Also for classification algorithms you can use Weka as it provides up to 100 different classifiers.

PCA or Principal Components Analysis for identifying the most important variables in you dataset. There are many studies that use PCA for data analysis but this is one of the papers that applies it in an original way : Eigenbehaviors: identifying structure in routine

Collaborative filtering for building recommendation systems. It is a problem of similarity matching. For example finding people who are similar to you in terms of the products they have liked or have purchased or finding products that are similar with respect to set of attributes. In this later case an "item based" recomendation algorithm is performed

Dimension reduction algorithm for reducing dimensions of a data set .It tries to take a large set of data and replace it with a smaller set of data that contains much of the important information in the larger set. For example you can use the reduced data set for undestanding it better and visualize it in 2D dimensions.

Association rules or co-occurrence grouping for market basket analysis. A common question in this case is : what items are commonly purchased together? For example, analyzing purchase records from a supermarket may uncover that tunna fish is purchased together with tooth paste frequently .

LDA algorithm for sentiment analysis and text mining but also for many other applications. Here a paper about it : Page on aaai.org

Dijkstra's Shortest Path for finding the shortest way from a node to another in a graph. Probably the most important algorithm in graph theory.

Link prediction to predict connections between data items, by suggesting that a link should exist, and estimating the strength of the link. For example in social networking it tries to guess if you and John share 15 friends, maybe you and John could be friens also in real life.

A good way to understand algorithms is to visualize them. For more on that try Algorithms and Data Fusion

Greg Skinner, 25+ years of software engineering, mostly in C and Perl, mostly in the BSD networking stack and Uni…

There are no such algorithms. A true software engineer knows how to look up an algorithm when it is needed and is open minded enough to consider several alternative algorithms when presented with a challenge. Yes you learn various algorithms in school but the most important thing you learn is the existence (or non-existence) of algorithms for various purposes and their relative advantages and drawbacks. A true software engineer (as opposed to a programmer) is more concerned about how to design an effective and appropriate software solution. This may involve research on solutions and algorithms, some of which may have been developed recently and not studied in school. When it comes time to write the code, (s)he may need to look up the details of a particular algorithm in order to make a final selection and refresh his/her memory on the details.

I had a question on this on my PhD final exam from Maurice Halstead, a very accomplished software engineer who joined academia after a long career in software design and development at Lockheed-Martin Corp. He asked me how I would solve certain problems and when I didn’t know solutions to all of them he asked what I should do about that. The wrong answer would have been to say i’d “wing it” or that I’d go back and memorize solutions to those problems. The correct answer was that I’d research the problems and look up solutions. His point was that nobody can or should rely only on memorization because there will always be new problems you haven’t solved before or recently. You have more important things to do than remember all the details. You need to know how to look up details when needed. Moreover, you won’t have seen all problems before and need to know how to research solutions rather than just diving in. This is one of the differences between a software engineer and a programmer.

Algorithm is a solution. It means, the work is done. We figured it out, made a library and moved on. Unless you have to work specifically on those tasks due to not having any library support, you will never write a tree, a list, or any kind of sorting in your life. Update: Thanks for all of the constructive comments! Perhaps I should have added a paragraph at the beginning to say: I don’t think they should necessarily know these algorithms by heart, but they should know that these algorithms exist, and know how to look them up and implement them. They should also know how these algorithms can affect the run-time complexity of their code. Most of the time in modern programming, we use code from libraries, and the software engineer can simply choose among several alternatives. For example, the C++ STL provides many container classes. The container can have a big impact on code performance. There are hundreds, if not thousands of libraries available to the modern software engineer. In that case, knowledge of the underlying algorithms can help them to make a better choice. In any case, even these very solved problems need to be written by someone, even if that someone is just writing a library to be used by others.

I wouldn’t memorize any algorithms, personally. You’ll probably end up memorizing a few through studying them, but that’s not the point. The point is to learn the techniques for designing algorithms that have worked well in the past. Study the algorithms to master the techniques they employ, not to memorize the algorithm itself - the reason being, anything you can memorize can be found in a library that has the algorithm already written for you and accessible to you through a simple function call. A good engineer needs to be able to deal with situations where a baked-in solution isn’t available.

I would suggest going through this text or something similar: http://educacioncreativa.org/OMI...

Once you get through that, if you are still interested in continuing your study of algorithms, you might dive into the problems the Big Data and massive parallel computation fields are facing - looks like you could get a good start using the PDFs in this course layout: CIS 700: algorithms for Big Data

The reason I suggest diving into Big Data and massively parallel algorithms after this is they will introduce new techniques not encountered in the first PDF that employ probability and statistics to increase efficiency - it will expand the way you think about algorithms.

If you are wanting to quiz/test people to make sure they can write algorithms, you’ll want to understand enough about your own business to understand which algorthmic problems you’re most likely to encounter (graph traversals, sorting, probabilistic programming…) and come up with problems that test candidates’ abilities to employ the techniques for designing algorithms specific to your business.

Patrick Minton, Software Engineer since the 90s
Written Aug 15, 2016

Are you asking the ones you need to know to pass an interview to get a job? Or the ones you need to know because you’ll use them every day?

I agree with those who’ve answered that you need to know the principles, not the specifics. Because honestly, you probably don’t need to know stuff like ‘Finding an optimal hash for a static hash table’ because you’re probably not going to implement your own. And if you know algorithmic principles, you can probably guess.

But since I’m guessing you are new to algorithms, I’d say that what you need to know is how and why things work in the languages/libraries that you use. Because that’s how you avoid writing inefficient code.

Here’s a classic example. Many interviews ask the candidate to de-dupe an array, and I see something like this frequently, and it’s a red flag:

  1. const deDupe = (inputArray) => {
  2.   let outputArray = []
  3.   inputArray.forEach((element) => {
  4.     if (!outputArray.indexOf(element)) {
  5.       outputArray.push(element)
  6.     }
  7.   })
  8.  
  9.   return outputArray
  10. }

Many junior developers treat line 4 like an O(n) operation, because they you don’t really know what indexOf is doing, and it looks like “one step”. But it’s really iterating through the list, so it’s O(n). Which means that this function is O(n^2), which is very inefficient. This won’t matter for small lists, but the problem with O(n^2) functions is that nobody notices the performance problem until suddenly somebody starts using deDupe() on arrays with hundreds of thousands of entries. Or, and this is the real scary scenario, they start using deDupe() on hundreds of thousands of arrays.

Many commands that you use every day (like shift or unshift) are similar. And that’s without even mentioning libraries. Do you know how .where() in a Backbone Collection (or a Ruby ActiveRecord model) works?

Understanding how your stuff works is really important.

To be a competent software engineer I think understanding the principles of data structures, fundamental algorithms, and asymptotic time and space complexity are very important. I've been a systems level software designer/developer for more than 30 years and these things continue to be my guiding principles.

In my own experience, mastery takes years and years of solving various computing problems, experimentation by trying different structures/algorithms, and especially by learning to apply your own variations to common patterns. Personally, I would never reduce this to the "10 that you must know by heart".

I'm a commercial software developer at SAS Institute and have been there for almost 32 years. I work on mostly lower-level, proprietary infrastructure components for our products. This is fairly specialized stuff with a steep learning curve. Everything has to scale massively and is subject to extensive testing. There is a lot of code to handle error recovery and high-availability/failover to take into account.

One could come in with a strong academic background in programming, data structures/algorithms, etc and still be challenged, even overwhelmed on many of our teams. In contrast, I witnessed an English major learn advanced programming by moving up through the ranks to eventually exceed many who came in with a BS in computer science. For about 20 years she was the brains behind an advanced and very powerful macro language. I knew the original designer (now deceased) of that language who was a bona fide genius. I don't think he had a formal computer science background but was brilliant in mathematics and also conversant in many subjects.
